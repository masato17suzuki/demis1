## Demis Layer1とは

**Demis Layer1**は、雇用主が求める人材像を5軸（IQ・学歴・文化資本・熱意・体力）のスライダーで設定し、その水準に合致する人材をAIが物語付きで生成する雇用主向けエンジンです。

さらに1000人の求職者データとのマッチングをシグモイド積モデルで高速計算し、採用圏内・圏外を瞬時に判定します。

## 対象ユーザーと課題

### 対象ユーザー
- 採用基準の妥当性をデータで検証したい人事担当者
- 求人条件を変えたときに候補者がどれだけ変化するかシミュレーションしたい企業

### 課題
1. **求人条件が厳しすぎるか甘すぎるかわからない**：感覚で条件を決めているため、候補者が集まらない原因が特定できない
2. **LLMに1000人分の判定をさせると壊れる**：コンテキスト超過・API費用爆発・ハルシネーションが発生する
3. **求人票の質が属人的**：担当者によって求人票の出来が変わる

### ソリューション
- 5軸スライダーで条件を設定し、AIが「その水準で最も輝く人材」を5人ずつ生成（ガチャ機能）
- 1000人との一括マッチングはCPU上のシグモイド積で高速処理（API費用ゼロ）
- 「AIアドバイスを元に求人票作成」「AIアドバイスを無視して求人票作成」の二択で、人間が最終判断を握る

## 実装上の工夫

### 1. LLMは「翻訳係」、計算は数式

最大の工夫は**マッチング計算にLLMを使わない**ことです。

1000人の求職者と求人条件を照合する100万通りの計算をLLMに任せると、莫大なAPI費用と処理時間がかかります。Demis Layer1では：

- **LLMの仕事**：人材プロファイルの生成と求人票のテキスト作成（1回約$0.01）
- **数式の仕事**：1000人のスコア計算（CPU上、ミリ秒、$0）

### 2. シグモイド積モデル（足切りの数式化）

```python
S = sp(iq) × sp(edu) × sp(cc) × sp(dcc) × sp(fit)
```

各能力値のスコアを**掛け算**します。1軸でも閾値を大きく下回れば全体が0になる。これは現実の「どれだけ優秀でも体力が致命的に足りなければ不採用」という足切り構造を数式で表現しています。加算モデルではこの現実を表現できません。

### 3. 市場賃金の自動算出

```python
W = W_base × max(0.5, 1.0 + Σ α_d(s_d - 5))
α = {IQ:0.04, 学歴:0.05, 文化資本:0.02, 熱意:0.02, 体力:0.01}
```

学歴係数（0.05）がIQ係数（0.04）より高い。日本の賃金構造の実測値に基づいています。スライダーを動かすと提示すべき適正賃金がリアルタイムで変化します。

### 4. 求人票作成の二択

「AIアドバイスを元に作成」と「AIアドバイスを無視して作成」の2ボタンを並べました。

- **アドバイスあり**：市場の人材供給を考慮し、現実的な条件に調整された求人票を生成
- **アドバイスなし**：雇用主の条件をそのまま求人票に反映。AIの修正提案を一切加えない

判断を常に人間に委ねる設計です。

## プロンプトの工夫

### 人材ガチャ（5人ずつ生成）

スライダーの設定値をプロンプトに変換する際、数値をそのまま渡すのではなく、人間が読める説明文に変換しています。

例：IQスライダー=7 → 「やや高い(110-115)」

これによりLLMが数値の意味を正確に理解し、リアルな人物像を生成します。出力はJSON形式を強制し、名前・年齢・ヘッドライン・ストーリー・強み・本音・感情の7要素を必ず含めます。パース失敗時はリトライ処理でエラーを防いでいます。

### シミュレーションモード

最下部の「シミュレーションモード」では、現在のスライダー設定に基づく求人1000件をJSON生成できます。3つのモード：

- **日本型統計**：日本の人口統計に基づく現実的な分布
- **ランダム**：完全均一分布（比較用）
- **今回のレベル**：現在のスライダー設定を中心とした分布

これらのJSONはE層（中央制御エンジン）に転送し、1000人×1000人のマッチングシミュレーションに使用します。

## まとめ

Demis Layer1は「AIに全部考えさせない」設計により、以下を実現しました。

- **高速**：1000人のマッチングがミリ秒で完了
- **低コスト**：マッチング計算のAPI費用がゼロ
- **正確**：シグモイド積による足切りが現実の採用判断を忠実に再現
- **透明**：全ての判定根拠が数式で追跡可能（説明可能AI）
